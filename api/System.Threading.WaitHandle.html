Class WaitHandle

Encapsulates operating system–specific objects that wait for exclusive access to shared resources.

Inheritance
Object
WaitHandle
Namespace:System.Threading
Assembly:System.Runtime.dll
Syntax
public abstract class WaitHandle : IDisposable
Remarks

The WaitHandle class encapsulates Win32 synchronization handles, and is used to represent all synchronization objects in the runtime that allow multiple wait operations. For a comparison of wait handles with other synchronization objects, see Overview of Synchronization Primitives.

The WaitHandle class itself is abstract. Classes derived from WaitHandle define a signaling mechanism to indicate taking or releasing access to a shared resource, but they use the inherited WaitHandle methods to block while waiting for access to shared resources. The classes derived from WaitHandle include:

  • The Mutex class. See Mutexes.

  • The EventWaitHandle class and its derived classes, AutoResetEvent and ManualResetEvent. See EventWaitHandle, AutoResetEvent, CountdownEvent, ManualResetEvent.

  • The Semaphore class. See Semaphore and SemaphoreSlim.

Threads can block on an individual wait handle by calling the instance method WaitOne(), which is inherited by classes derived from WaitHandle.

The derived classes of WaitHandle differ in their thread affinity. Event wait handles (EventWaitHandle, AutoResetEvent, and ManualResetEvent) and semaphores do not have thread affinity; any thread can signal an event wait handle or semaphore. Mutexes, on the other hand, do have thread affinity; the thread that owns a mutex must release it, and an exception is thrown if a thread calls the ReleaseMutex() method on a mutex that it does not own.

Because the WaitHandle class derives from System.MarshalByRefObject, these classes can be used to synchronize the activities of threads across application domain boundaries.

In addition to its derived classes, the WaitHandle class has a number of static methods that block a thread until one or more synchronization objects receive a signal.. These include:

  • System.Threading.WaitHandle.SignalAndWait(System.Threading.WaitHandle,System.Threading.WaitHandle), which allows a thread to signal one wait handle and immediately wait on another.

  • WaitAll(WaitHandle[]), which allows a thread to wait until all the wait handles in an array receive a signal.

  • WaitAny(WaitHandle[]), which allows a thread to wait until any one of a specified set of wait handles has been signaled .

The overloads of these methods provide timeout intervals for abandoning the wait, and the opportunity to exit a synchronization context before entering the wait, allowing other threads to use the synchronization context.

This type implements the IDisposable interface. When you have finished using the type or a type derived from it, you should dispose of it either directly or indirectly. To dispose of the type directly, call its System.Threading.WaitHandle.Close method in a try/catch block. To dispose of it indirectly, use a language construct such as using (in C#) or Using (in Visual Basic). For more information, see the “Using an Object that Implements IDisposable” section in the IDisposable interface topic.

WaitHandle implements the Dispose() pattern. See Dispose Pattern. When you derive from WaitHandle, use the System.Threading.WaitHandle.SafeWaitHandle property to store your native handle operating system handle. You do not need to override the protected Dispose(Boolean) method unless you use additional unmanaged resources.

WaitAll(WaitHandle[])

Waits for all the elements in the specified array to receive a signal.

Remarks

AbandonedMutexException is new in the .NET Framework version 2.0. In previous versions, the method returns true when a mutex is abandoned. An abandoned mutex often indicates a serious coding error. In the case of a system-wide mutex, it might indicate that an application has been terminated abruptly (for example, by using Windows Task Manager). The exception contains information useful for debugging.

The method returns when all the handles are signaled. On some implementations, if more than 64 handles are passed, a NotSupportedException is thrown. If the array contains duplicates, the call fails with a System.DuplicateWaitObjectException.

The method is not supported on threads that have STAThreadAttribute.

Calling this method overload is equivalent to calling the System.Threading.WaitHandle.WaitAll(System.Threading.WaitHandle[],System.Int32,System.Boolean) method overload and specifying -1 (or Infinite) for millisecondsTimeoutand true for exitContext.

WaitAny(WaitHandle[])

Waits for any of the elements in the specified array to receive a signal.

Remarks

AbandonedMutexException is new in the .NET Framework version 2.0. In previous versions, the method returns true if the wait completes because a mutex is abandoned. An abandoned mutex often indicates a serious coding error. In the case of a system-wide mutex, it might indicate that an application has been terminated abruptly (for example, by using Windows Task Manager). The exception contains information useful for debugging.

The method throws an AbandonedMutexException only when the wait completes because of an abandoned mutex. If waitHandles contains a released mutex with a lower index number than the abandoned mutex, the method completes normally and the exception is not thrown.

In versions of the .NET Framework earlier than version 2.0, if a thread exits or aborts without explicitly releasing a Mutex, and that Mutex is at index 0 (zero) in a WaitAny array on another thread, the index returned by WaitAny is 128 instead of 0.

This method returns when any handle is signaled. If more than one object becomes signaled during the call, the return value is the array index of the signaled object with the smallest index value of all the signaled objects. On some implementations, if more that 64 handles are passed, a NotSupportedException is thrown.

Calling this method overload is equivalent to calling the System.Threading.WaitHandle.WaitAny(System.Threading.WaitHandle[],System.Int32,System.Boolean) method overload and specifying -1 (or Infinite) for millisecondsTimeoutand true for exitContext.

WaitOne()

Blocks the current thread until the current WaitHandle receives a signal.

Remarks

AbandonedMutexException is new in the .NET Framework version 2.0. In previous versions, the method returns true when a mutex is abandoned. An abandoned mutex often indicates a serious coding error. In the case of a system-wide mutex, it might indicate that an application has been terminated abruptly (for example, by using Windows Task Manager). The exception contains information useful for debugging.

The caller of this method blocks indefinitely until the current instance receives a signal. Use this method to block until a WaitHandle receives a signal from another thread, such as is generated when an asynchronous operation completes. For more information, see the IAsyncResult interface.

Calling this method overload is equivalent to calling the System.Threading.WaitHandle.WaitOne(System.Int32,System.Boolean) method overload and specifying -1 or Infinite for the first parameter and false for the second parameter.
